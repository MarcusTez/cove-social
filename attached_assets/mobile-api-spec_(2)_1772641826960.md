## Environments

The API is available in two fully isolated environments. Use Development for building and testing your integration, then switch to Production for the live app.

| | Development | Production |
|---|---|---|
| **Base URL** | `https://e4af2c56-d31e-4016-b6f4-4605cbfaf1bf-00-9jq2nkbugewe.worf.replit.dev/api/mobile` | `https://www.cove-social.com/api/mobile` |
| **Database** | Development database (safe to create test users, test data) | Production database (real user data) |
| **Stripe** | Stripe test mode (use Stripe test cards) | Stripe live mode (real payments) |
| **Purpose** | Integration testing, development, QA | Live mobile app |

### Key Differences

- **Completely isolated**: Development and Production have separate databases, separate user accounts, and separate JWT signing keys. Nothing is shared between them.
- **Tokens are not interchangeable**: An access token or refresh token obtained from the Development environment will not work in Production, and vice versa. Each environment uses its own JWT secret.
- **Test users**: You can create test accounts in the Development environment without affecting real user data. Test users need to go through the same onboarding + invite + password setup flow as real users.
- **Stripe payments**: Development uses Stripe test mode. Use [Stripe test card numbers](https://docs.stripe.com/testing#cards) (e.g., `4242 4242 4242 4242`) for testing subscriptions.

### Recommended Workflow

1. **Develop and test** against the Development base URL.
2. **Verify** all flows work end-to-end (login, profile updates, matches, etc.) with test data.
3. **Switch** the base URL in your app config to Production for release builds.
4. **Test production** with a real account before submitting to app stores.

### Environment Configuration (Mobile App)

We recommend storing the base URL as a configuration variable in your mobile app so it can be easily switched:

```
// Development
API_BASE_URL = "https://e4af2c56-d31e-4016-b6f4-4605cbfaf1bf-00-9jq2nkbugewe.worf.replit.dev/api/mobile"

// Production
API_BASE_URL = "https://www.cove-social.com/api/mobile"
```

All endpoint paths in this document are relative to the base URL. For example, the login endpoint is `POST {API_BASE_URL}/auth/login`.

## Overview

All endpoints are prefixed with `/api/mobile`. The API uses JSON for request and response bodies. All authenticated endpoints require a Bearer token in the `Authorization` header.

### Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Content-Type` | Yes | `application/json` |
| `Authorization` | For protected routes | `Bearer <access_token>` |

---

## Authentication

The API uses JWT-based authentication with short-lived access tokens and long-lived refresh tokens.

### Flow

1. **Login** with email + password to receive an `accessToken` (1 hour) and `refreshToken` (30 days).
2. Use the `accessToken` as a Bearer token for all protected endpoints.
3. When the access token expires (you receive a `401` with `code: "TOKEN_EXPIRED"`), call the **Refresh** endpoint with your `refreshToken` to get a new pair.
4. On **Logout**, send the refresh token to invalidate it server-side.

### Token Storage (Mobile Best Practice)

- Store `accessToken` in memory (not persistent storage).
- Store `refreshToken` in secure storage (Keychain on iOS, EncryptedSharedPreferences on Android).

## Error Format

All errors return a consistent JSON structure:

```json
{
  "error": "Human-readable error message",
  "code": "OPTIONAL_ERROR_CODE",
  "details": []
}
```

### Common HTTP Status Codes

| Status | Meaning |
|--------|---------|
| `200` | Success |
| `201` | Created |
| `400` | Bad Request (validation error) |
| `401` | Unauthorized (missing/expired token) |
| `403` | Forbidden (account inactive or invalid token) |
| `404` | Not Found |
| `409` | Conflict (duplicate resource) |
| `500` | Internal Server Error |

### Waitlist

#### POST `/api/mobile/waitlist`

Register interest and join the waitlist. This is **not** account registration — it adds the user to the waitlist. Users will receive an invite code through the existing invite flow when a spot opens.

A verification email is sent to confirm the email address. The user must click the link in the email to verify.

**Auth Required:** No

**Request Body:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `email` | string | Yes | Email address |
| `gender` | string | No | Gender |
| `city` | string | No | City |

**Example Request:**

```json
{
  "email": "jane@example.com",
  "gender": "Woman",
  "city": "London"
}
```

**Success Response (201):**

```json
{
  "id": "abc-123-def",
  "email": "jane@example.com",
  "gender": "Woman",
  "city": "London",
  "emailVerified": false,
  "createdAt": "2026-03-04T10:00:00.000Z"
}
```

**Error Responses:**

| Status | Body | When |
|--------|------|------|
| `400` | `{"error": "Invalid request data", "details": [...]}` | Missing or invalid email |
| `409` | `{"error": "This email is already on the waitlist"}` | Email already registered |

**Notes:**
- After signing up, the user receives a verification email with a confirmation link.
- Once verified, they enter the drip email sequence and will eventually receive an invite to become a member.
- This endpoint does **not** create an account or grant login access. Login is only available after a user has completed onboarding and set a password through the invite flow.

#### POST `/api/mobile/auth/refresh`

Exchange a refresh token for a new access token + refresh token pair. The old refresh token is revoked.

**Auth Required:** No

**Request Body:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `refreshToken` | string | Yes | The current refresh token |

**Example Request:**

```json
{
  "refreshToken": "a1b2c3d4e5f6..."
}
```

**Success Response (200):**

```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiIs...",
  "refreshToken": "newtoken123456...",
  "expiresIn": 3600
}
```

**Error Responses:**

| Status | Body | When |
|--------|------|------|
| `400` | `{"error": "Refresh token is required"}` | Missing field |
| `401` | `{"error": "Invalid or expired refresh token"}` | Token revoked/expired |

```json

  "success": true

### Matching Preferences

#### PATCH `/api/mobile/matching-preferences`

**Auth Required:** Yes

**Request Body (at least one field required):**

| Field | Type | Required | Description |

{

}
```

```json
{
  "matchingOptIn": false,
  "pauseUntil": null
}
```

## Password Requirements

Passwords must meet all of the following:
- Minimum 8 characters
- At least one uppercase letter
- At least one lowercase letter
- At least one number
- At least one special character

---

## Rate Limiting

Currently no rate limiting is enforced. This may be added in a future version. Mobile clients should implement reasonable request throttling (e.g., no more than 60 requests per minute).

---

## Versioning

The API is currently at v1. All endpoints are under `/api/mobile/`. If breaking changes are introduced, a new prefix (e.g., `/api/mobile/v2/`) will be used and the v1 endpoints will remain available during a deprecation period.
